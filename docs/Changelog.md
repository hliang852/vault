# Changelog

Dated log of changes made to this repo and why. Newest entries first.

## 2026-07-06 (5) -- Part 2: axis-tag clustering, community-detection cross-check, new weighted features

Implements the core Part 2 redesign called for in `Claude.md`'s product vision: precedents should group into **overlapping clusters**, feeding a future descriptive scenario-playbook UI (Part 3, not built yet). This session builds the data layer for that; no UI consumes it yet.

- **Added `data/Japan.csv` columns (134 -> 137)**: `multi_jurisdiction` (`num_regulators >= 2`), `notes_flags_precedent_setting` (keyword flag on `notes_raw`: "first"/"template"/"precedent"/"proof-of-concept"/"landmark"/"signature" -- 7/62 true), `price_was_bumped_bool` (a clean boolean fix: the source `price_was_bumped` column is an object column mixing the strings `'True'`/`'False'`/`'Unclear/TBV'` and `NaN`, and the existing `bool(value)` pattern used elsewhere in `precedent_engine.py` would incorrectly treat `'Unclear/TBV'` and `NaN`-as-string as truthy; `price_was_bumped_bool` is `True` only where the source is the literal string `'True'` -- 7/62 true, matching the ~7/62 estimate). All three mechanically derived, no new facts. Documented in `docs/Japan_User_Guide.md`.
- **Added `src/cluster/axes.py`** -- a new module, factored out of (not copy-pasted from) `notebooks/exploratory_analysis.py`'s Lenses 1/2/3/4/5/6/8. Exposes `compute_axes(row, activist_columns)` returning 7 descriptive per-case axes (consent, price-discovery type, regulatory friction, precedent-value text, instigator identity, lock-up signal count, activist signature), plus a ported `jaccard`/`top_cooccurring_pairs` helper for double-counting checks. These axes are the intended multi-membership clustering signal -- a case naturally carries several at once.
- **Extended `src/cluster/precedent_engine.py`**:
  - 7 new `BOOLEAN_FEATURES` entries, weights grounded in each feature's real true-rate in the 62-row corpus (not guessed) -- see `docs/Architecture.md` for the full table and rationale. `timeline_post_fiea_2026_amendment` contributes 0 to every score today (0/62 true; the amendment postdates this corpus) and will only start mattering once the corpus grows past 2026-05-01.
  - Deliberately did **not** score `instigator_type`/`lockup_signal_count`/`activist_signature` as features -- each recombines columns already scored independently, so scoring the composite too would double-count.
  - Attached each node's `axes` dict (from the new module) and a `short_name` (compact display label -- strips parentheticals and trailing corporate-suffix stopwords from `target_full_name`, falls back to the first 3 words if empty).
  - Added a secondary overlapping-community cross-check via `networkx.algorithms.community.k_clique_communities` on a full (untrimmed) pairwise graph at the existing `MIN_SCORE=1.5`. At `k=3` this is **degenerate on the current 62-case corpus** -- a single community containing all 62 nodes -- so every node's `community_ids` is currently `[0]` and not yet informative; kept as-is per the reviewed plan (try k=3 first, report if degenerate rather than silently re-tuning) and flagged in `To-do.md`.
  - Added `networkx` to `requirements.txt`.
- **Viewer**: changed the always-visible node label in `viewer/Japan_Precedent_Constellation.html` from `d.id` to `d.short_name` -- `deal_id` remains visible in the detail panel and search, unchanged.
- **Ran an empirical Jaccard double-counting check** (Task 4 of this session) across the full boolean feature set now scored by `precedent_engine.py` (all pre-existing `BOOLEAN_FEATURES`/`activist_*` plus the 7 new features this session), min support 3. Beyond the already-acknowledged pairs, found several **unexpected** high-overlap (Jaccard > 0.6) pairs not previously flagged: `timeline_post_meti_2023_guideline` x `timeline_post_tse_reform_2023` (0.93), `has_JFTC` x `has_DOJ_FTC_HSR` (0.89), `structure_has_TOB` x `has_JFTC` (0.80), `has_DOJ_FTC_HSR` x `timeline_post_tse_reform_2023` (0.71), `structure_has_TOB` x `has_DOJ_FTC_HSR` (0.71), `toehold_present_flag` x `timeline_post_tse_reform_2023` (0.68), `has_DOJ_FTC_HSR` x `timeline_post_meti_2023_guideline` (0.67), `toehold_present_flag` x `timeline_post_meti_2023_guideline` (0.67), `has_JFTC` x `timeline_post_tse_reform_2023` (0.63). No weights were changed in response -- weights are a maintainer judgment call per `Claude.md`; logged for review in `docs/To-do.md`.
- **Caught and fixed a bug during post-implementation review**: the viewer's `const DATA = {...}` object is hand-embedded JavaScript (see `Architecture.md`'s "known gap"), not loaded from `output/precedent_graph_data.json` at runtime. The node-label fix (`d.id` -> `d.short_name`) was applied to the rendering code, but the embedded `DATA` blob itself was never re-synced after `precedent_engine.py` was re-run -- so every node's `short_name` would have rendered as `undefined` in the actual browser. Fixed by re-embedding the freshly regenerated JSON into the HTML file (same manual copy-paste process already documented as a Part 3 gap). Verified post-fix: the embedded `DATA.nodes[0]` now carries `short_name`/`axes`/`community_ids`, matching `output/precedent_graph_data.json`.
